## О проекте

**GPS Tracker** — мобильное приложение для отслеживания пробежек, написанное на Python с использованием Kivy.

Приложение получает GPS-координаты устройства, отображает текущее положение на карте, строит пройденный маршрут и сохраняет информацию о тренировках в локальную SQLite-базу данных.

Проект разрабатывается с упором на:

* разделение пользовательского интерфейса и бизнес-логики;
* изоляцию Android-зависимого кода;
* локальное хранение пользовательских данных;
* возможность дальнейшего расширения функциональности;
* поддержку фонового GPS-трекинга;
* понятную и масштабируемую архитектуру.

В приложении реализованы:

## Архитектура

Проект разделён на несколько слоёв:

```text
UI
 │
 ▼
Services
 │
 ├──────────────► Domain
 │
 ├──────────────► Database
 │
 └──────────────► Platform API
```

### UI

Содержит:

* экраны приложения;
* кнопки;
* нижнюю навигацию;
* карту;
* маркеры;
* элементы отображения маршрута.

UI не должен напрямую работать с SQL-запросами или Android API. Он передаёт пользовательские действия сервисному слою и отображает полученный результат.

### Services

Сервисный слой содержит основную бизнес-логику:

* запуск и остановку тренировки;
* обработку GPS-координат;
* управление состоянием трекинга;
* расчёт параметров пробежки;
* передачу данных между UI и репозиторием;
* сохранение результатов тренировки.

### Domain

Содержит модели предметной области, которые не зависят от Kivy, SQLAlchemy или Android API.

Например, `GPSPoint` описывает одну GPS-точку маршрута.

### Database

Слой базы данных отвечает за:

* создание SQLAlchemy Engine;
* настройку сессий;
* описание ORM-моделей;
* сохранение пробежек;
* сохранение GPS-точек;
* получение истории тренировок;
* удаление и очистку данных.

### Platform API

Содержит код, который зависит от конкретной операционной системы.

Android-интеграция изолирована от UI и основной бизнес-логики. Благодаря этому платформенный код можно изменять отдельно от остальных частей приложения.

## Модель данных

В базе данных используются две основные сущности.

### Run

Хранит общую информацию о тренировке:

* идентификатор пробежки;
* время начала;
* время завершения;
* продолжительность;
* пройденную дистанцию;
* среднюю скорость.

### RunPoint

Хранит отдельную GPS-точку маршрута:

* идентификатор точки;
* идентификатор пробежки;
* широту;
* долготу;
* высоту;
* скорость;
* направление движения;
* точность GPS;
* время получения координаты.

Связь между таблицами:

```text
Run 1 ───────────── N RunPoint
```

Одна пробежка может содержать множество GPS-точек.

## Технические решения

### Repository Pattern

Взаимодействие с базой данных вынесено в `RunRepository`.

UI и сервисы не должны самостоятельно создавать SQL-запросы. Они используют публичные методы репозитория.

Это позволяет:

* уменьшить связанность компонентов;
* централизовать работу с базой;
* упростить тестирование;
* заменить реальный репозиторий тестовой реализацией.

### Dependency Injection

Общие сервисы и репозитории создаются на уровне корневого контейнера приложения и передаются экранам через конструкторы.

Это позволяет:

* не создавать несколько экземпляров одного сервиса;
* контролировать жизненный цикл зависимостей;
* избегать глобальных переменных;
* заменять реальные зависимости тестовыми.

### Callback-based GPS API

`GPSService` не зависит от конкретного экрана.

Полученные координаты и изменения состояния передаются через callback-функции:

```text
GPSService
    │
    ├── on_location(GPSPoint)
    └── on_status(str)
```

Благодаря этому один GPS-сервис можно использовать независимо от конкретной реализации UI.

### Platform abstraction

Android-зависимый код находится в каталоге `platform_api`.

Основная бизнес-логика не должна напрямую зависеть от:

* Java-классов;
* Android Service;
* Android Context;
* системных разрешений;
* конкретной версии Android API.

## Ограничения

* GPS-функциональность зависит от оборудования устройства.
* Точность маршрута зависит от качества GPS-сигнала.
* В помещении координаты могут определяться неточно.
* На настольных операционных системах реальный GPS может быть недоступен.
* Android может ограничивать работу приложения в фоне.
* Режим энергосбережения может остановить получение координат.
* Текущая APK-сборка предназначена для тестирования.
* Приложение пока не опубликовано в Google Play.

## Безопасность данных

Все тренировки и настройки сохраняются локально на устройстве.

Текущая версия приложения:

* не требует регистрации;
* не отправляет историю тренировок на внешний сервер;
* не использует облачное хранение;
* не передаёт GPS-маршруты другим пользователям.

При удалении приложения локальные данные могут быть удалены операционной системой.
