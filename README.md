## О проекте

**GPS Tracker** — pet-проект мобильного приложения для отслеживания пробежек.

Приложение получает координаты устройства, отображает текущее положение на карте, записывает GPS-маршрут и сохраняет информацию о тренировках в локальную базу данных.

Проект разрабатывается с упором на:

* разделение пользовательского интерфейса и бизнес-логики;
* изоляцию платформозависимого Android-кода;
* локальное хранение данных;
* возможность дальнейшего расширения функциональности;
* поддержку фонового GPS-трекинга;
* понятную и масштабируемую структуру проекта.

## Основные возможности

* получение GPS-координат устройства;
* отображение текущего положения на карте;
* запуск и остановка записи тренировки;
* построение пройденного маршрута;
* сохранение GPS-точек в SQLite;
* сохранение продолжительности и дистанции пробежки;
* просмотр истории тренировок;
* просмотр подробной информации о выбранной пробежке;
* хранение пользовательских настроек;
* переключение между экранами через нижнюю навигационную панель;
* подготовленная платформенная интеграция с Android.

## Архитектура

Проект разделён на несколько независимых слоёв:

```text
UI → Services → Domain → Database
          ↓
    Platform API
```

### UI

Содержит экраны, кнопки, навигацию, карту и другие визуальные компоненты.

UI не должен напрямую управлять базой данных или Android API. Он передаёт пользовательские действия сервисному слою и отображает полученный результат.

### Services

Содержит основную бизнес-логику приложения:

* запуск и остановка тренировки;
* обработка GPS-координат;
* управление состоянием трекинга;
* расчёт и сохранение данных пробежки;
* взаимодействие между интерфейсом и репозиторием.

### Domain

Содержит модели предметной области, которые не зависят от Kivy, SQLAlchemy или Android API.

Например, `GPSPoint` описывает одну GPS-точку маршрута.

### Database

Отвечает за:

* настройку SQLAlchemy;
* ORM-модели;
* создание сессий;
* сохранение пробежек;
* сохранение GPS-точек;
* получение истории тренировок.

### Platform API

Изолирует код, зависящий от конкретной операционной системы.

Android-интеграция вынесена из UI и бизнес-логики, благодаря чему платформенный код можно изменять независимо от остальных частей приложения.

## Модель данных

Основные сущности базы данных:

### Run

Хранит общую информацию о пробежке:

* время начала;
* время завершения;
* продолжительность;
* пройденную дистанцию;
* среднюю скорость.

### RunPoint

Хранит отдельную точку маршрута:

* широту;
* долготу;
* высоту;
* скорость;
* направление движения;
* точность GPS;
* время получения координаты.

Одна пробежка может содержать множество GPS-точек.

```text
Run 1 ─────────── N RunPoint
```

## Технические решения

### Repository Pattern

Доступ к базе данных вынесен в отдельный репозиторий.

UI и сервисы не формируют SQL-запросы напрямую, а используют методы `RunRepository`.

Это уменьшает связанность компонентов и упрощает тестирование.

### Dependency Injection

Общие сервисы и репозитории создаются на уровне корневого контейнера приложения и передаются экранам через конструкторы.

Такой подход позволяет:

* не создавать несколько экземпляров одного сервиса;
* контролировать жизненный цикл зависимостей;
* заменять реальные реализации тестовыми;
* избегать глобальных переменных.

### Callback-based GPS API

`GPSService` не зависит от конкретного экрана.

Полученные координаты и изменения состояния передаются через callback-функции.

Это позволяет использовать GPS-сервис в разных экранах и не связывать его напрямую с Kivy UI.

### Platform abstraction

Android-специфичный код находится в `platform_api`.

Бизнес-логика не должна напрямую знать о Java-классах, Android Service или системных разрешениях.
